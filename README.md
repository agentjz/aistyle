# README

这个项目最核心的目标，其实说起来很简单，但真正要把它写清楚、做清楚，并不容易。这个目标就是：人如何借助 AI，同时也通过学习 AI 的做事方式，让自己的成长变得更快、更稳，而且少走很多原本可以不走的弯路。

这里说的“更快”，不是那种口号式的更快，也不是把 AI 想象成万能按钮。这个项目真正关心的，是更现实、更具体的东西。比如，一个人面对陌生领域的时候，能不能更快找到入口；一个人在资料很多、概念很杂的时候，能不能更快抓住主线；一个人在脑子发散、事情堆积、方向模糊的时候，能不能更快把混乱压成结构。换句话说，这个项目想研究的，不是“AI 替人活”，而是“AI 帮人减少无谓的慢”。

所以这个项目从一开始就不打算往几个方向滑过去。它不追求神秘感，不想把 AI 写得像某种神迹；它也不讨论底层实现细节，因为这个项目不是给工程师内部看技术设计的；它更不想把一切都写成工作流、KPI、效率工具那套语言。它更想讨论的是学习、成长、生活方向、兴趣社区、判断力这些东西，讨论人在真实生活里怎样借助 AI 活得更明白一点、更少内耗一点。

根目录现在只保留几个最重要的入口。`README.md` 是总入口，告诉任何一个新开的对话，这个项目到底在干什么；[文档规范.md](C:/Users/Administrator/Desktop/thuglife/文档规范.md) 是必须服从的写作原则，防止项目越写越偏；[提问引导.md](C:/Users/Administrator/Desktop/thuglife/提问引导.md) 只放当前这一轮最值得回答的问题；`历史对话/` 只记录发生过什么；`主题沉淀/` 则只保存整理之后、值得反复读的成果。

如果每次新开一个对话，最稳的阅读顺序是这样的。先读 `README.md`，先重新把这个项目的总目标抓回来；接着读 [文档规范.md](C:/Users/Administrator/Desktop/thuglife/文档规范.md)，免得后面写着写着又滑回提纲味、程序员味或者空泛味；然后再读 [提问引导.md](C:/Users/Administrator/Desktop/thuglife/提问引导.md)，确认这一轮真正的问题是什么。读完这三样以后，再去看 [01_当前状态.md](C:/Users/Administrator/Desktop/thuglife/主题沉淀/00_项目总览/01_当前状态.md)、[02_索引.md](C:/Users/Administrator/Desktop/thuglife/主题沉淀/00_项目总览/02_索引.md)、[03_更新日志.md](C:/Users/Administrator/Desktop/thuglife/主题沉淀/00_项目总览/03_更新日志.md)，最后再进入各个主题文件夹。

目前这个项目里最核心的主题主要有三条。第一条是 [01_AI加速成长](C:/Users/Administrator/Desktop/thuglife/主题沉淀/01_AI加速成长)，它讨论的是 AI 为什么能帮助人少走弯路，以及人怎样把这种“加速”真正变成自己的成长方式。第二条是 [02_GPT从问题到交付](C:/Users/Administrator/Desktop/thuglife/主题沉淀/02_GPT从问题到交付)，它讨论的是 GPT 在收到问题之后，通常是怎样一步一步把问题推进到交付结果的。第三条是 [03_自媒体账号实践](C:/Users/Administrator/Desktop/thuglife/主题沉淀/03_自媒体账号实践)，它开始把这些理解往现实作品里落，也就是把“AI 如何加速人的成长”做成一个内容账号。

这个项目的更新规则也很简单，但要长期坚持。每轮对话结束以后，AI 至少要做三件事：第一，把这一轮发生了什么整理进 `历史对话/`，这样以后不会丢上下文；第二，更新 [提问引导.md](C:/Users/Administrator/Desktop/thuglife/提问引导.md)，确保下一轮不会还在回答过期问题；第三，如果这轮已经形成了稳定成果，就把它写进 `主题沉淀/`，而不是只停留在聊天窗口里。
